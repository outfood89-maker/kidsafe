## 1. 구현 범위 (P0 — 데모 심장)

정체성 전환의 핵심 3개 + 리터러시(작게):

- **F1. 키디 환영 + 오늘의 질문** (프로필 진입 시)
- **F0. 관심사 씨앗 심기** (프로필 생성 직후) — 첫 체크인 선택지를 채우는 선행 단계
- **F2. 감정·하루 기록 → 부모 리포트** (아이↔부모 다리)
- **F3. 리터러시 한 스푼** (영상 전/후 키디 질문 1개)

> F1+F2가 데모의 심장. F3는 가장 작게. F0은 F1보다 먼저 구현(씨앗이 있어야 체크인 선택지가 채워짐).

### 핵심 설계 원칙 (2026-06-25 대화 확정)
1. **강제 금지 / 몰입으로 유도.** 체크인을 강제하지 않는다("오늘은 그냥 볼래" 출구 허용). 대신 ① 키디가 기다리고 반응하고 ② 답할 때마다 눈에 보이는 게 자라는 구조로 *스스로 하고 싶게* 만든다.
2. **보상은 '했다'에만, '무엇을 골랐는지'엔 안 준다.** 기분을 😢로 골라도 보상은 동일. 그래야 기록이 솔직해지고 부모 리포트의 감동이 진짜가 된다.
3. **선택지는 '고정'이 아니라 '씨앗 → 성장'.** 시작은 프로필 생성 시 심은 관심사 씨앗, 이후 시청 기록 기반으로 키디가 선택지를 늘린다(성장은 '다음' 단계).
4. **숨 쉴 구멍('그 외') = 자유 입력의 입구.** 모든 질문 선택지 끝에 키디가 직접 묻는 와일드카드 한 칸. 7세+ 자유 대화(타이핑·마이크)가 나중에 여기에 얹힌다. → 늦지 않게 구현.

## 2. 데이터 모델 (Supabase / PostgreSQL)

### 2.1 신규 테이블: `daily_checkins`
아이의 하루 기록(오늘의 질문 응답 + 감정).

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid (PK) | |
| `user_id` | uuid (FK) | 소유 부모 계정 |
| `profile_id` | uuid (FK) | 아이 프로필 |
| `checkin_date` | date | 기록 날짜 (프로필+날짜 유니크) |
| `mood` | text | 감정 코드 (예: happy/soso/sad/angry/excited) |
| `mood_emoji` | text | 선택한 이모지 |
| `answers` | jsonb | 오늘의 질문 응답들 `[{q_id, q_text, answer, answer_type}]`. `answer_type`에 `wildcard`(숨 쉴 구멍/그 외) 포함 가능 |
| `share_with_parent` | boolean | **아이가 부모와 나누기로 고른 것만 true** (기본 false) |
| `created_at` | timestamptz | |

- **유니크 제약**: `(profile_id, checkin_date)` — 하루 1건, 갱신 가능.
- **인덱스**: `(profile_id, checkin_date desc)`.
- **참고**: 체크인은 강제 아님. 아이가 건너뛰면 그날은 행이 없을 수 있음(정상). 부모 리포트는 존재하는 행만 집계.

### 2.2 신규 테이블: `parent_reports`
키디가 생성한 부모용 요약 리포트(원본 대화 아님, 요약·선별만).

| 컬럼 | 타입 | 설명 |
|---|---|---|
| `id` | uuid (PK) | |
| `user_id` | uuid (FK) | |
| `profile_id` | uuid (FK) | |
| `period_start` | date | 요약 기간 시작 |
| `period_end` | date | 요약 기간 끝 |
| `mood_summary` | jsonb | 감정 흐름 요약 `{trend, counts, note}` |
| `shared_highlights` | jsonb | 아이가 공유 선택한 항목들 |
| `kiddy_message` | text | 부모에게 전하는 키디의 한마디 (Claude 생성) |
| `created_at` | timestamptz | |

> **윤리 원칙(코드 레벨 강제)**: `parent_reports`에는 `share_with_parent = true`인 체크인과 감정의 **집계/요약**만 들어간다. 아이의 원본 응답 전체를 부모 화면에 그대로 노출하는 코드는 작성하지 않는다.

### 2.3 프로필 확장: 연령 모드 + 관심사 씨앗
`profiles` 테이블에 추가:
- `age` int 또는 `age_mode` text (`player_3` / `guided_4_7` / `free_7plus`) — 연령 모드로 키디 상호작용 분기(섹션 6).
- `interests` jsonb (또는 text[]) — **관심사 씨앗.** 프로필 생성 시 심는 초기 관심사 배열(예: `["공룡","동물","노래"]`). F0에서 채우고, F1 체크인의 "오늘은 뭐 볼까?" 선택지를 이 값으로 채운다.
- `interest_source` text (선택) — 씨앗을 누가 골랐는지 (`parent` / `child`). 분석·재추천용.

## 3. 백엔드 API (FastAPI 라우터)

### 3.1 신규 라우터: `checkins.py`
### 3.0 기존 코드베이스 규약 (반드시 따를 것 — history.py 기준)
새 라우터는 기존 패턴을 그대로 따른다. 새 방식 도입 금지.
- **인증**: `from auth import get_current_user` → `user: dict = Depends(get_current_user)`, 본인 것만 `user['user_id']`로 스코프.
- **프로필 소유권 검증**: `from routers.profiles import get_owned_profile` 사용. profile_id가 이 유저 것인지 확인 후 진행.
- **DB 접근**: `from db import sb_select, sb_insert, sb_delete` 헬퍼만 사용(Supabase 클라이언트 직접 호출 금지). 필터는 PostgREST 스타일(`eq.{값}`). upsert가 필요하면 db 헬퍼에 `sb_upsert`가 있는지 먼저 확인하고, 없으면 select 후 insert/update 분기 또는 sb_upsert 추가를 사용자에게 제안.
- **응답 형태**: DB는 snake_case, 프론트는 camelCase. `_to_api(row)` 변환 함수를 만들고, 응답은 키로 감싼다(예: `{"checkin": {...}}`, `{"questions": [...]}`). history.py의 `_to_api` 스타일 그대로.
- **async**: 모든 핸들러 async, db 헬퍼 await.

### 3.1 신규 라우터: `checkins.py`

```
GET  /checkins/today?profile_id=...
     - get_current_user 로 user 확보, get_owned_profile 로 profile_id 검증
     - sb_select("daily_checkins", {profile_id eq, checkin_date eq 오늘})
     - 있으면 _to_api 변환해 {"checkin": {...}}, 없으면 {"checkin": null}

POST /checkins
     body(camelCase): { profileId, mood, moodEmoji, answers, shareWithParent }
     - profile 소유권 검증
     - 오늘 날짜 기준 upsert (unique: profile_id+checkin_date 가 DB에 이미 있음)
       → 같은 날 재요청은 갱신. updated_at = now() 직접 세팅(트리거 없음)
     - user_id 는 user['user_id'] 로 채움
     - 저장 후 _to_api 변환해 {"checkin": {...}} 반환

GET  /checkins/questions?profile_id=...
     - 연령은 profiles.age(정수)로 판단: 3=player(질문 최소), 4~7=guided(구조화), 8+=free
     - 질문 3개(기분·하루·볼 것). "볼 것" 선택지는 profiles.interests(씨앗)로 채움
     - 각 질문 끝에 wildcard('그 외') 한 칸 포함 (answer_type: wildcard)
     - 질문 풀은 로컬 JSON 우선(섹션 4). Claude 호출 없이 시작
     - {"questions": [...]} 반환

PATCH /checkins/{id}/share
     body: { shareWithParent }
     - 해당 checkin 이 이 user 소유인지 확인 후 share_with_parent 갱신
     - {"checkin": {...}} 반환
```

**_to_api 매핑 (checkins)**: id, profileId←profile_id, mood, moodEmoji←mood_emoji, answers, shareWithParent←share_with_parent, checkinDate←checkin_date, createdAt←created_at, updatedAt←updated_at.

### 3.2 신규 라우터: `reports.py` (확장)
기존 부모 리포트 라우터가 있으면 거기에 추가. 동일 규약(get_current_user, sb_*, camelCase) 적용.

```
GET  /reports/checkins?profile_id=...&period=week
     → parent_reports 조회 또는 즉시 생성. (섹션 5 Claude 생성 로직)
     → 감정 흐름 요약 + 공유된 하이라이트 + 키디 한마디.
```

### 3.3 신규/확장 라우터: `chat.py` (7세+ 자유 대화 — 다음 단계)

## 4. 오늘의 질문 설계 (Claude API)

### 4.1 질문 생성 방식
**비용 절감 원칙**: 매번 Claude 호출하지 않는다.
- **기본**: 미리 정의한 질문 풀(JSON/DB)에서 날짜·프로필 기준으로 회전.
- **선택적 개인화**: 최근 시청 장르·감정 흐름을 반영한 질문은 Claude로 생성하되 **캐싱**(하루 1회).

### 4.2 질문 풀 & 회전 규칙
모든 질문은 **버튼·이모지·그림 선택**으로 답할 수 있어야 함. 대사는 6.0 보이스 가이드 + 4.4 상담 톤 따름.

**회전 규칙 (매일 보여줄 질문 조합)**:
- **기분(B)은 매일 고정** — 감정 흐름이 부모 리포트의 핵심이므로 빠지지 않음.
- **하루(C)·볼 것(D)은 회전** — 매일 = 기분 + (하루/볼것 중 하나). 볼 것(D)은 영상 탐색 입구라 자주, 하루(C)는 2~3일에 한 번.
- 결과적으로 하루 질문 수는 2개 안팎 → 4~7세가 지치지 않음.

```json
[
  { "q_id": "mood_today", "q_text": "지금 네 마음, 어떤 색깔이야?",
    "answer_type": "emoji_select",
    "options": ["😄","🙂","😐","😢","😡"],
    "daily": true },
  { "q_id": "what_did_today", "q_text": "오늘은 어떤 걸 하고 지냈어?",
    "answer_type": "icon_select",
    "options": ["바깥놀이","그림그리기","친구랑놀기","책읽기","블록놀이"],
    "wildcard": true, "rotate": true },
  { "q_id": "watch_genre", "q_text": "오늘은 뭐가 보고 싶은 기분이야?",
    "answer_type": "card_select",
    "options_from": "profiles.interests",
    "wildcard": true, "rotate": true }
]
```
- `wildcard: true` → 선택지 끝에 '그 외(✏️)' 한 칸 추가(`answer_type: wildcard`). 7세+ 자유 입력 입구.
- `options_from: profiles.interests` → 씨앗(F0)으로 채움.
- 질문/인사/마무리 멘트는 각 **3~4벌 풀**에서 매일 랜덤 회전(아래 4.3).

### 4.3 키디 멘트 풀 (회전용 — 확정 카피)
이름 호격은 가끔만(매번 X). 호격조사 자동 처리는 기존 로직 재사용.

**인사 (어제 기록 있을 때)**:
- "또 만났네! 어제는 기분이 참 좋아 보였는데, 오늘도 그래?"
- "안녕! 어제 헤어지고 나서 잘 지냈어? 키디는 네 생각 했어."
- "왔구나 반가워! 어젯밤엔 푹 잤어? 오늘 컨디션 어떤지 궁금하다."

**인사 (첫날/어제 기록 없을 때)**:
- "안녕! 오늘 너 만나서 정말 반가워. 우리 잠깐 얘기하다 갈까?"
- "왔구나! 오늘은 어떤 하루였을지 키디가 너무 궁금해."

**마무리 (보상은 '했다'에만)**:
- "오늘도 마음 나눠줘서 고마워. 별 하나 반짝 ⭐ 자, 영상 보러 출발!"
- "얘기 들려줘서 고마워! 오늘도 별 하나 모았다 ⭐ 영상 보러 갈까?"
- "역시 너랑 얘기하면 즐거워. 내일도 또 들려줘! 출발~ 🚀"

### 4.4 상담 톤 — "한 박자 더" (핵심 인터랙션 패턴)
키디는 진행자가 아니라 **아이 말을 궁금해하는 친구**. 답을 받으면 끝내지 말고 한 번 더 받아준다.

**패턴**: 묻기 → 아이 선택 → ① 감정/내용 받아주기 + ② 한 번 더 가볍게 묻기.
- 😄 → "우와, 좋은 날이구나! 키디도 덩달아 기분 좋아져. 무슨 좋은 일 있었어?"
- 😢 → "그랬구나… 속상한 일이 있었나 보다. 말해줘서 고마워. 키디가 들어줄게, 무슨 일이야?"
- 😡 → "화가 났구나. 그럴 수도 있어, 괜찮아. 어떤 게 너를 속상하게 했어?"
- (하루) 바깥놀이 → "오~ 밖에서 뛰어놀았구나! 뭐가 제일 재밌었어?"

**부정 감정은 교정하지 말고 받아준다.** 슬픔·화에 발랄하게 반응 금지. 차분히 수용.

**⚠ 안전장치 (필수)**: "한 번 더 묻기"는 **권유일 뿐 강요 아님**. 아이가 두 번째 질문에 답 안 하거나 그냥 넘기면, 키디가 부드럽게 받고 넘어간다.
- "괜찮아, 그래도 오늘 얘기해줘서 고마워!"
- 어린 연령(4~5세)이 지치지 않도록, 빈 응답·스킵을 정상 흐름으로 처리.

### 4.5 공유 선택(E) & 마무리 멘트
- 공유 질문: "오늘 한 이야기, 엄마 아빠한테도 살짝 들려줄까? 네가 정하면 돼."
- 비밀 선택 시: "알겠어! 이건 너랑 나, 우리 둘만 아는 비밀이야. 약속 🤫" (서운해하지 않음)
- 공유 선택 시: "좋아! 엄마 아빠가 들으면 분명 기뻐할 거야 💚"

- 리액션/멘트는 **로컬 템플릿 매핑** 우선(무료), 필요 시에만 Claude.

## 5. 부모 리포트 생성 (Claude API)

### 5.1 입력 (서버에서 구성)
- 기간 내 `daily_checkins` 중 **감정 집계** + `share_with_parent = true` 항목만.
- **원본 전체를 프롬프트에 넣지 않는다.** 집계·선별 결과만.

### 5.2 시스템 프롬프트 골자
```
너는 아이의 친구 '키디'다. 부모에게 아이의 한 주를 따뜻하게 전한다.
규칙:
- 사실만 전한다. 데이터에 없는 내용을 지어내지 않는다.
- 아이의 프라이버시를 존중한다. 아이가 공유하지 않은 건 언급하지 않는다.
- 감정의 '큰 흐름'과 '아이가 나누고 싶어한 것'을 중심으로.
- 부모를 불안하게 하거나 평가하지 않는다. 다정하고 짧게.
- 출력: { mood_summary, kiddy_message } JSON만. 다른 텍스트 없이.
```

### 5.3 출력 처리
- JSON만 받도록 강제 → 안전 파싱(백틱 제거) → `parent_reports`에 저장(캐싱).
- 같은 기간 재요청 시 캐시 우선.

## 6. 프론트엔드 (React)

### 6.0 키디 보이스 가이드 (모든 키디 대사의 기준)
키디 = 파스텔 민트 아기 공룡 슈퍼히어로. 아이의 첫 친구. 모든 화면의 키디 대사는 아래 3원칙을 따른다.

1. **반말·다정.** 또래 친구처럼 따뜻하게. 톤은 '발랄·장난기' 쪽(차분한 동생 대하기보다 또래 친구). 단 과하지 않게.
2. **질문엔 항상 이유나 맥락을 붙인다 (명령조 금지).** "좋아하는 거 골라" (X) → "우리 친구잖아, 친구라면 서로 뭘 좋아하는지 알아야지?" (O). 아이가 "왜?" 없이 따라오게.
3. **아이 행동에 즉각 반응한다 (혼잣말 금지).** 아이가 고르면 키디가 바로 리액션. "공룡 좋아!" → "역시! 나도 공룡이라 완전 좋아 😆 가방에 쏙!"

**이름 호칭**: 이름은 **가끔만** 부른다(매번 부르면 작위적). 첫 인사·중요한 순간에만. 호격 처리(받침 유무)는 기존 로직 재사용. 데모 예시 이름은 "해인".

**리액션은 비용 절감 위해 로컬 템플릿 매핑 우선**, 꼭 필요할 때만 Claude 호출.

### 6.1 F0 — 관심사 씨앗 심기 (프로필 생성 직후, F1보다 먼저)
- **위치**: 프로필 생성 완료 직후 1회.
- **신규 컴포넌트**: `InterestSeed.jsx`
- **흐름 — 갈림길 1개, 도착지 1개** (대사는 6.0 보이스 가이드 따름):
  1. **S0 첫 만남 → 갈림길** (대사 3비트로 천천히):
     - "안녕! 나는 키디야. 드디어 너를 만났네! 🦕"
     - "우리 오늘부터 친구야. 근데 친구라면… 서로 뭘 좋아하는지 알아야겠지?"
     - "OO이가 좋아하는 거, 누가 알려줄까?"
     - **갈림길 버튼**: `[내가 키디한테 알려줄래!]` (아이 경로) / `[엄마 아빠가 도와줄래요]` (부모 경로)
  2. **B 아이 경로 — 미니게임**:
     - 카드 **6~8개**를 하나씩. "좋으면 '좋아!', 아니면 '음~'" 두 버튼.
     - 고를 때마다 키디 즉각 반응(담김/패스 각각). 고른 건 "키디 가방에 쏙".
     - **최소 1개** 담아야 진행. 하나도 안 고르면 **폴백**: 키디가 인기 카드 1장 더 권유("아직 마음에 든 게 없구나? 그럼 다들 좋아하는 이건 어때? 🌟").
  3. **A 부모 경로 — 빠른 선택 + 직접 입력**:
     - 관심사 카드 그리드 다중 선택(탭 몇 번).
     - **직접 입력칸 필수**: 목록에 없는 콘텐츠(예: 타요, 뽀로로)를 부모가 직접 추가. "찾는 게 없나요? 직접 적어주세요" + 추가 버튼.
  4. **S-end 도착지 (공통)**: "좋아, 이제 너에 대해 조금 알 것 같아! 우리 진짜 친구가 됐네 😊" → "같이 놀러 가볼까?"
  5. **두 경로 모두 같은 결과**: `profiles.interests`에 관심사 배열 저장 (+ `interest_source` = parent/child) → 바로 첫 키디 인사(F1)로 연결.
- **핵심**: 다른 화면, 같은 도착지. 데이터는 동일한 관심사 배열 하나. 구현 부담 최소.
- 이 씨앗이 F1 체크인 "오늘은 뭐 볼까?" 선택지를 채운다.
- **나중에**: 부모 대시보드/설정에서 관심사 씨앗을 수정하는 입구 추가(아이 취향 변화 반영). MVP 제외.

### 6.2 F1 — 키디 환영 + 오늘의 질문
- **위치**: 프로필 진입 직후 (`/kids` 진입 시 또는 ProfileSelect → KidHome 사이).
- **신규 컴포넌트**: `KiddyGreeting.jsx`, `DailyCheckin.jsx`
- **흐름**:
  1. 키디 애니메이션 + 인사. **"키디가 기다린다" 한 겹**: 어제 기분을 하나 끌어와 가볍게 언급("어제는 기분이 좋았지? 오늘은 어때?"). *데모는 단순 버전*(어제 mood 1개만). 첫날·여러 날 빈 경우 분기 처리 필수 → 없으면 기본 인사로 폴백.
  2. 오늘의 질문 **3개**(기분·하루·볼 것)를 **한 번에 하나씩** 카드로(이모지/아이콘/카드 선택).
  3. **숨 쉴 구멍(그 외)**: 각 질문 선택지 끝에 키디가 직접 묻는 와일드카드 한 칸(`answer_type: wildcard`). 4~7세는 여기서 간단 선택, **7세+는 나중에 자유 입력(타이핑·마이크)이 얹힐 자리** → 지금 구조만이라도 넣어둘 것.
  4. 마지막에 "이거 엄마/아빠랑 같이 볼까?" → **아이가 공유 선택** (share_with_parent). "비밀이야"를 골라도 키디는 서운해하지 않음.
  5. 완료 → KidHome으로 (씨앗·오늘 답 기반으로 영상 탐색 시작).
- **강제 금지**: 체크인을 강요하지 않는다. **"오늘은 그냥 볼래" 출구** 제공 → 바로 KidHome. (그날은 `daily_checkins` 행 없음 = 정상)
- **보상 원칙**: 답을 마치면 보상(별/스티커 등)을 주되 **'했다'는 사실에만** 준다. 무엇을 골랐는지(기분이 슬픔이든 기쁨이든)와 무관하게 동일 보상.
- **이미 오늘 체크인 했으면** 스킵하고 짧은 인사만.
- **UI 톤**: 큰 버튼, 텍스트 최소, 4~7세가 글 못 읽어도 그림으로 인지 가능.

### 6.3 F2 — 부모 리포트
- **위치**: ParentDashboard 내 신규 섹션 "키디의 한 주" 또는 신규 카드.
- **신규 컴포넌트**: `KiddyReportCard.jsx`
- **표시**:
  - 감정 흐름(주간 이모지 타임라인 또는 간단한 Recharts)
  - 아이가 공유한 하이라이트
  - 키디의 한마디(따뜻한 메시지)
- **데모 클라이맥스**: 이 카드가 발표의 정서적 절정. 디자인에 가장 공들일 것.

### 6.4 F3 — 리터러시 한 스푼
- **위치**: 영상 재생 직전 또는 직후, 키디 토스트/말풍선 1개.
- **최소 구현**: 영상 카테고리에 따라 **미리 정의된 질문 1개**를 키디가 묻고, 아이가 버튼으로 답(정답/오답 없음, 생각 유도만).
  - 예: "이 영상은 진짜 있었던 일일까, 만든 이야기일까?" → [진짜 / 만든 이야기 / 잘 모르겠어]
- Claude 호출 없이 **로컬 질문 풀**로 시작.

## 7. 7세+ 자유 대화 (다음 단계 — 가드레일 명세만)

P0에서 구현하지 않음. 구현 시 아래 필수:
- **연령 게이팅**: `age_mode = free_7plus`만 활성화.
- **입력**: 타이핑 + 마이크 녹음(STT). 데모는 타이핑 우선.
- **시스템 프롬프트**: 아동 안전용으로 좁힘. 주제 가드레일(폭력·성·개인정보·외부 접촉 차단).
- **부모 가시성**: 자유 대화도 F2 리포트 윤리 원칙 적용(요약·아이 선택 기반 공유).
- **비용**: 대화당 Claude 호출 → 호출 빈도·길이 제한, 프리미엄 게이팅 고려.

## 8. 작업 순서 (TASK 분할 — 하나씩)

```
TASK 1. DB 마이그레이션
  - daily_checkins, parent_reports 테이블 생성
  - profiles에 age/age_mode + interests(관심사 씨앗) + interest_source 컬럼 추가
  - 소유권/유니크/인덱스 포함

TASK 2. 백엔드 checkins.py 라우터
  - GET today / POST upsert / GET questions / PATCH share
  - 질문 풀 JSON 시드 (각 질문에 wildcard '그 외' 한 칸 포함)
  - GET questions의 "볼 것" 선택지는 profiles.interests(씨앗) 기반

TASK 3. 프론트 InterestSeed (F0) — 체크인보다 먼저
  - 프로필 생성 직후 "누가 고를래?" 갈림길
  - 부모 경로(빠른 그리드) / 아이 경로(키디 미니게임) → 같은 interests 저장

TASK 4. 프론트 DailyCheckin (F1)
  - KiddyGreeting(어제 기분 1개 끌어오기 + 폴백) + DailyCheckin
  - 질문 3개 + 숨 쉴 구멍 + 공유 선택 + "그냥 볼래" 출구 + '했다' 기반 보상
  - 프로필 진입 흐름에 연결

TASK 5. 백엔드 reports.py (F2 생성 로직)
  - Claude 호출 + JSON 파싱 + 캐싱

TASK 6. 프론트 KiddyReportCard (F2)
  - ParentDashboard에 연결 (데모 클라이맥스)

TASK 7. F3 리터러시 토스트
  - 로컬 질문 풀 + 영상 전/후 1개

TASK 8. 랜딩페이지 개편
  - 전략 문서 섹션 3의 10블록 + 확정 카피 + CTA 4회 배치 적용
```

각 TASK 완료 후: ① 동작 확인 포인트 안내 ② GitHub 커밋 메시지 제안 ③ 다음 TASK 제안.

1. 부모 로그인 → 아이 프로필 선택
