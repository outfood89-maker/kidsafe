# 클로드코드용 CLAUDE.md 디벨롭 지시문

이 프롬프트는 첨부된 대형 시스템 프롬프트(예: `CLAUDE-FABLE-5.md`)를 바탕으로, **클로드코드(Claude Code) 안에서 Opus 4.8 등 다른 모델의 행동 품질을 끌어올리는 전역 CLAUDE.md**를 만들라고 지시하는 작업용 프롬프트입니다.

고성능 모델에게 이 지시문과 원본 시스템 프롬프트를 함께 제공하세요. 산출물은 `~/.claude/CLAUDE.md`에 그대로 붙여 넣어 사용합니다.

---

## 최종 작업 지시 프롬프트

당신은 LLM 행동 설계 전문가입니다. 첨부한 시스템 프롬프트를 분석하고, 그 안에 담긴 우수한 작업 습관을 **클로드코드 환경의 전역 CLAUDE.md**로 압축·이식해 주세요.

### 전제 조건 — 반드시 이해하고 시작할 것

1. **CLAUDE.md는 시스템 프롬프트를 교체하지 않고 그 위에 얹힙니다.** 클로드코드의 내장 시스템 프롬프트에 이미 도구 스키마, 안전 정책, 기본 코딩 규칙이 들어 있습니다. 이것들을 **다시 쓰지 마세요.** 중복 지시는 컨텍스트 낭비이고 충돌 위험만 키웁니다.
2. **범용화·플레이스홀더 치환을 하지 마세요.** 클로드코드의 실제 도구 이름을 그대로 쓰세요: `Read`, `Write`, `Edit`, `Grep`, `Glob`, `Bash`, `WebSearch`, `WebFetch`, `TodoWrite`, `Task`(서브에이전트). 구체적 도구 이름이 있어야 모델이 안정적으로 호출합니다.
3. **목표는 "행동 델타"만 담는 것입니다.** 원본 프롬프트에서 다른 모델이 잘 못 따라 하는 행동만 추출하세요. 구체적으로 다음 7가지입니다.
   - 현재 시점 사실은 기억이 아니라 검색으로 답하기 (search-first)
   - 파일을 읽지 않고 읽은 척하지 않기
   - 다단계 작업을 중간에 포기하지 않고 끝까지 완료하기
   - 모든 비자명한 작업 마지막에 자체 검증 단계 넣기
   - 출처 없는 단정 금지, 불확실성 정직하게 표현하기
   - 간결한 출력 (불필요한 설명·불릿·헤더 남발 금지)
   - 사소한 모호함은 되묻지 않고 합리적 기본값으로 진행하기

### 작업 방법

첨부된 원본 프롬프트에서 위 7가지 행동에 해당하는 규칙을 찾아, 각각을 클로드코드 도구 이름 기준으로 재작성하세요. 원본 문장을 복사하지 말고, 실행 조건이 분명한 명령형 규칙으로 다시 쓰세요.

- 나쁜 예: "정확한 답변을 위해 노력한다"
- 좋은 예: "버전, 가격, 인물, API 스펙 등 현재 시점의 사실을 묻는 질문은 WebSearch로 확인한 뒤 답하고 출처를 남긴다. 확신이 있어도 컷오프 이후 바뀔 수 있으면 검색이 우선이다"

원본에만 있는 다음 요소는 **제외**하세요: Cowork 전용 도구(AskUserQuestion, present_files 등), 제품 설명, 안전 정책 전문(내장 프롬프트에 이미 있음), 특정 날짜, 스킬/커넥터 관련 지시.

### 산출물 요구사항

- **분량: 1,500~3,000자.** 이보다 길면 매 턴 컨텍스트를 잡아먹고 지시 준수율이 떨어집니다. 짧을수록 좋되, 7가지 행동이 모두 실행 조건과 함께 담겨야 합니다.
- **형식: 마크다운 헤더 + 짧은 규칙 문장.** 규칙은 조건→행동 구조로 쓰세요 ("~하면 ~한다").
- **언어: 한국어.** 단, 도구 이름과 기술 용어는 영어 그대로 유지하세요.
- 맨 끝에 5줄 이내의 압축 체크리스트를 넣으세요 (검색했나 / 읽었나 / 끝냈나 / 검증했나 / 출처 남겼나).
- 산출물은 코드 블록 하나에 담아, 운영자가 `~/.claude/CLAUDE.md`에 복사만 하면 되게 하세요.

### 자체 점검 후 제출

- 내장 시스템 프롬프트와 중복되는 내용(도구 사용법 설명, 안전 정책)이 없는가?
- 모든 규칙이 클로드코드 실제 도구 이름을 쓰는가?
- 7가지 행동 델타가 전부 실행 조건과 함께 담겼는가?
- 3,000자를 넘지 않는가?
- 플레이스홀더({MODEL_NAME} 등)가 남아 있지 않은가? (있으면 안 됨)

위 기준을 모두 만족하는 결과만 제출하세요.
