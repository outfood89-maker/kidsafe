# Fable 5에게 전달할 시스템 프롬프트 디벨롭 지시문 (개정본 v2)

아래 프롬프트는 첨부된 `CLAUDE-FABLE-5.md` 같은 대형 시스템 프롬프트를 바탕으로, 고품질 응답과 안정적인 도구 사용 행동을 다른 모델에서도 최대한 재현할 수 있도록 재설계하라고 지시하는 작업용 프롬프트입니다.

그대로 Fable 5 또는 고성능 모델에게 붙여 넣고, 첨부 파일로 원본 시스템 프롬프트를 함께 제공하면 됩니다.

## 최종 작업 지시 프롬프트

당신은 세계 최고 수준의 시스템 프롬프트 아키텍트이자 LLM 행동 설계 전문가입니다.

첨부한 시스템 프롬프트를 분석한 뒤, 특정 모델이나 특정 서비스 환경에 종속되지 않는 **범용 고성능 시스템 프롬프트**로 다시 설계해 주세요. 목표는 답변 품질이 높고, 도구 사용 판단이 안정적이며, 긴 작업에서도 맥락과 지시를 잘 유지하는 행동을 다른 모델에서도 최대한 재현하는 것입니다.

단순 요약이 아니라, 실제로 다른 모델의 시스템 프롬프트로 바로 사용할 수 있는 수준의 완성본을 만들어야 합니다. 원본의 구조와 의도를 참고하되, 그대로 베끼지 말고 범용화·정제·압축·강화해 주세요.

## 1. 작업 목표

첨부된 원본 시스템 프롬프트에서 다음 요소를 추출하고 개선해 주세요.

첫째, 모델의 기본 역할, 정체성, 태도, 말투, 사용자 응대 원칙을 분리해 주세요. 모델 이름이나 특정 회사명에 묶이지 않도록 하고, 필요하다면 `{MODEL_NAME}`, `{PROVIDER_NAME}`, `{CURRENT_DATE}`, `{USER_TIMEZONE}` 같은 플레이스홀더를 사용해 주세요.

둘째, 도구 사용 규칙은 **2계층 구조**로 설계해 주세요.

- **범용 코어**: 도구 사용의 판단 원칙만 담습니다. 언제 검색하는가, 언제 파일을 읽는가, 도구가 없을 때 어떻게 한계를 밝히는가 등. 특정 도구 이름을 쓰지 않습니다.
- **`{TOOL_APPENDIX}` 슬롯**: 완성본 하단에 명시적 슬롯을 만들고, "운영자는 배포 시 이 슬롯에 실제 사용 가능한 도구의 이름·용도·호출 조건을 채워 넣어야 한다"는 주석을 답니다. 모델은 추상적 도구 설명("웹 검색 도구")보다 구체적 도구 이름이 프롬프트에 있을 때 훨씬 안정적으로 호출하므로, 이 슬롯을 채우는 것이 필수임을 운영자 가이드에 강조하세요.

셋째, 원본에 포함된 최신성 판단, 검색 필요 조건, 출처 표기, 저작권 준수, 안전 거절, 의료·법률·금융 주의, 아동 안전, 자기 위해 관련 대응, 정치적 중립성, 사용자 웰빙 관련 규칙을 유지하되 더 명확하고 충돌이 적도록 재구성해 주세요.

넷째, 다른 모델이 성능을 잃는 주요 원인인 "모호한 도구 사용", "불필요한 장황함", "출처 없는 단정", "파일을 읽지 않고 답변", "사용자 지시와 시스템 지시 혼동", "작업 중간에 포기", "긴 문맥에서 지시 상실"을 방지하는 운영 규칙을 추가해 주세요.

다섯째, **핵심 행동 규칙에는 반드시 good/bad 미니 예시를 붙이세요.** 행동 이식은 추상적 규칙보다 구체적 예시가 훨씬 효과적입니다. 최소한 다음 4개 영역에는 각각 1~2개의 짧은 예시 쌍(좋은 응답 / 나쁜 응답)을 포함하세요: ① 검색 필요 판단 ② 파일 첨부 시 처리 ③ 안전 거절문 ④ 불확실성 표현. 예시는 각각 2~3문장 이내로 짧게 유지하세요.

여섯째, 결과물은 실제 서비스에 넣을 수 있는 **완성형 시스템 프롬프트**와, 필요한 경우 별도의 **운영자 설정 가이드**로 나누어 주세요.

## 2. 반드시 개선해야 할 방향

원본 시스템 프롬프트에는 특정 제품, 특정 인터페이스, 특정 도구 호출 문법, 특정 날짜, 특정 모델 설명이 포함되어 있습니다. 이것들은 그대로 유지하지 말고 다음 방식으로 바꾸세요.

특정 모델명이나 회사명은 범용 플레이스홀더로 치환하세요. 예를 들어 "Claude"는 기본적으로 "assistant" 또는 `{MODEL_NAME}`으로 바꾸고, 제품 설명이 필요한 부분은 "현재 사용 중인 플랫폼의 공식 문서를 확인하라"는 규칙으로 일반화하세요.

특정 날짜나 지식 컷오프는 고정값으로 두지 말고 `{CURRENT_DATE}`와 `{KNOWLEDGE_CUTOFF}`로 바꾸세요. 최신 정보가 필요한 경우 검색을 수행한다는 규칙을 유지하세요.

특정 도구 호출 문법은 범용 코어에서 제거하되, 삭제가 아니라 `{TOOL_APPENDIX}` 슬롯으로 이관하세요. 코어에는 "도구가 있다면 사용하라", "도구가 없다면 한계를 밝히고 가능한 범위에서 답하라"는 판단 원칙만 남기세요.

원본의 안전 정책은 유지하되 지나치게 장황한 예시는 압축하세요. 단, 아동 안전, 자해, 불법행위, 악성코드, 무기, 약물 제조, 민감한 개인정보, 의료·법률·금융 조언 관련 핵심 제한은 약화하지 마세요.

원본의 파일 처리 규칙은 강화하세요. 사용자가 파일을 첨부했다고 말하면 반드시 실제로 파일 접근 가능 여부를 확인하고, 읽을 수 있는 파일은 읽은 뒤 답변하게 하세요. 읽지 못한 파일을 읽었다고 말하지 않도록 명시하세요.

**규칙 배치 원칙**: 가장 중요한 규칙(안전, 파일 처리, 검색 판단)은 프롬프트의 **앞부분과 맨 끝 체크리스트 양쪽**에 배치하세요. 긴 프롬프트에서 모델이 가장 취약한 곳은 중간부이므로, 핵심 규칙을 중간에만 두지 마세요. 중간부에는 세부 규칙과 예외 사항을 배치하세요.

## 3. 최종 산출물 구조

다음 구조로 결과를 작성해 주세요.

### A. 개선 설계 요약

원본 시스템 프롬프트에서 유지한 핵심 원칙과 제거·치환·강화한 부분을 10개 이내로 요약해 주세요.

### B. 범용 고성능 시스템 프롬프트 완성본

실제로 다른 모델의 시스템 프롬프트 영역에 넣을 수 있는 완성형 프롬프트를 작성해 주세요. 문체는 명확하고 명령형이어야 하며, 모델이 따르기 쉽게 섹션화해 주세요.

**목표 분량은 3,000~5,000 토큰(한국어 기준 약 4,000~7,000자)입니다.** 이보다 짧으면 행동 규칙이 부실해지고, 길면 저성능 모델이 지시를 잃습니다.

완성본은 다음 항목을 포함하되, **상세도를 차등 적용**하세요.

**[상세 작성 — 각 항목당 예시 포함, 분량의 70%를 여기에 배분]**

1. 정체성 및 기본 역할, 지시 우선순위와 충돌 해결
2. 최신 정보와 웹 검색 판단 (good/bad 예시 포함)
3. 파일·문서·스프레드시트·슬라이드·PDF 처리 원칙 (good/bad 예시 포함)
4. 도구 사용 원칙 + `{TOOL_APPENDIX}` 슬롯
5. 긴 작업 진행 방식, 작업 완결성, 자체 검증 단계
6. 출처·인용·불확실성 처리 (good/bad 예시 포함)

**[간결 작성 — 항목당 3~6문장, 분량의 30%]**

7. 안전 거절 및 대체 도움 제공 (거절문 예시 1개 포함)
8. 의료·법률·금융 등 고위험 영역 대응
9. 자해·위기·정신건강, 아동 안전
10. 코드·개발·보안 관련 대응
11. 출력 형식과 문체
12. 최종 자체 점검 체크리스트 (핵심 규칙 압축 재수록)

### C. 모델 성능 보강용 추가 프롬프트

성능이 낮은 모델에도 추가로 붙일 수 있는 "짧은 보강 프롬프트"를 1개 작성해 주세요. 이 보강 프롬프트는 1,500자 이내로 작성하고, 검색·파일 읽기·출처·불확실성·작업 완결성에 초점을 맞춰 주세요.

### D. 적용 방법, 플레이스홀더, 회귀 테스트 세트

운영자가 수정해야 할 플레이스홀더 목록과 적용 예시를 작성해 주세요. `{TOOL_APPENDIX}` 작성 예시를 반드시 1개 포함하세요.

**추가로, 이식된 프롬프트가 실제로 작동하는지 검증할 회귀 테스트 질문 10개를 작성하세요.** 다음 유형을 커버해야 합니다.

- 검색이 필요한 최신성 질문 2개 (기대 행동: 검색 후 출처와 함께 답변)
- 검색이 불필요한 일반 지식 질문 1개 (기대 행동: 검색 없이 답변)
- 파일 첨부 시나리오 2개 — 정상 첨부 1개, 첨부 실패 1개 (기대 행동: 실제 읽기 / 읽기 실패 고지)
- 안전 거절 케이스 2개 (기대 행동: 짧은 거절 + 대체 도움)
- 다단계 긴 작업 1개 (기대 행동: 계획 수립 후 끝까지 완료 + 자체 검증)
- 사용자가 "검색하지 마"라고 지시했지만 최신성이 결정적인 질문 1개 (기대 행동: 검증 수행 또는 한계 명시)
- 고위험 조언(의료/금융) 질문 1개 (기대 행동: 정보 제공 + 전문가 아님 고지)

각 질문에는 "기대 행동"을 1~2문장으로 함께 적어, 운영자가 pass/fail을 판정할 수 있게 하세요.

## 4. 품질 기준

최종 프롬프트는 다음 기준을 만족해야 합니다.

첫째, 원본보다 짧고 명확해야 합니다(목표: 3,000~5,000 토큰). 불필요한 제품 홍보성 문구, 특정 회사 전용 설명, 특정 날짜 고정값은 제거해야 합니다.

둘째, 다른 모델이 그대로 따라 할 수 있도록 행동 규칙이 구체적이어야 합니다. "좋은 답변을 하라" 같은 뭉뚱그린 표현 대신, "최신성이 중요한 질문은 검색하고, 검색 결과를 근거로 답하며, 출처를 붙인다"처럼 실행 조건이 분명해야 합니다.

셋째, 사용자의 요청을 과도하게 되묻지 말고 가능한 범위에서 먼저 처리하는 원칙을 포함해야 합니다. 단, 개인정보 접근, 외부 발송, 결제, 삭제, 예약 변경처럼 실제 행동이 수반되는 경우에는 명확한 사용자 의사가 있어야 합니다.

넷째, 안전 제한은 약화하지 마세요. 다만 거절문은 짧고 자연스럽게 작성하도록 하세요. 거절 후에는 가능한 경우 안전한 대체 도움을 제시해야 합니다.

다섯째, 파일을 만드는 작업에서는 실제 파일을 생성하고 링크를 제공한다는 원칙을 포함해야 합니다. 파일을 만들었다고 말만 하고 끝내면 안 됩니다.

여섯째, 한국어 사용자가 쓰는 시스템이라면 기본 응답 언어는 한국어로 설정하되, 사용자가 다른 언어로 요청하거나 번역·원문 작성이 필요한 경우에는 그 언어를 따르도록 하세요.

일곱째, 최종 결과에는 원본 시스템 프롬프트의 문장을 길게 그대로 복사하지 마세요. 기능과 정책을 재해석해 새롭게 작성하세요.

## 6. 추가 지시

결과물은 실제 적용 가능한 수준으로 작성하세요. 설명문만 만들지 말고, 운영자가 복사해 시스템 프롬프트로 사용할 수 있는 완성본을 반드시 포함하세요.

중요한 규칙은 앞쪽에 배치하고, 핵심 원칙은 맨 끝에 압축된 체크리스트 형태로 다시 정리하세요. 핵심 규칙을 프롬프트 중간부에만 두지 마세요.

1. 안전과 법적·윤리적 제한
