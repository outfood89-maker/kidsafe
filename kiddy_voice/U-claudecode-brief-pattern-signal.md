# [Claude Code 작업 지시서] U — 감정 패턴 감지 최소 버전 (LLM 0, 결정적 계산)

> **왜:** D 기둥(발견)의 핵심 증거 — "키디는 매일의 안부를 모아 **흐름**을 본다". R 데모 시드(3주 mood 스토리: 평온→슬픔 상승→sad3+angry1)가 정확히 이 신호를 발화시키도록 설계돼 있음 (R §1 표).
> **설계 원칙:** 패턴 판정은 **서버 결정적 계산 (LLM 0)** — "사실은 코드가". care_signal(위기, 즉시)과 별개 층위: 이건 **흐름 관찰의 부드러운 신호**. 진단 단어·미래 예측·캐묻기 유도 금지.
> **✅ 카피 게이트 통과 (2026-07-03 팀장 검수 완료):** 임계값 승인+데이터 하한 추가, 배너 본문 원안 승인+서브라인 추가. 아래 §1·§3은 **확정본 — verbatim 구현, 임의 수정 금지.**
> **일정:** G0 게이트(7/9) 대상. R 최종 검증(패턴 신호 발화 확인)이 이 작업에 의존 → **T 다음, R 마무리 전.** (큐·의존 관계 팀장 승인)

---

## 1. 판정 규칙 (서버 — [reports.py `get_checkin_report`](../server/routers/reports.py#L529)에 추가)

- **데이터:** 최근 21일 체크인의 `mood`만 (기존 쿼리와 별도로 21일 범위 1회 조회, 또는 기간 확장 후 필터 — 구현 재량. answers 접근 금지).
- **주간 부정 카운트:** W2(21~15일 전) / W1(14~8일 전) / W0(최근 7일)별로 `mood in ("sad","angry")` 개수.
- **발화 조건 (팀장 확정):** `W2 < W1 < W0` (2주 연속 상승) **and** `W0 >= 3` **and** **데이터 하한: W0·W1 각각 체크인 4일 이상**.
  - 하한 이유(팀장 수정): 카운트 기반은 체크인 빈도와 기분이 섞임 — W1에 3일, W0에 7일 접속했다면 부정 증가가 기분 악화가 아니라 사용량 증가일 수 있음.
  - R 시드 검산: W2=0, W1=2, W0=4 + 매일 체크인(하한 통과) → 발화 ✓ / 노이즈: 0→1→2는 W0<3 미발화 ✓ / W1 체크인 3일이면 하한 미달 → 미발화 ✓.
- **응답 필드 (additive — spread 호환 유지):** `report.patternSignal = {"active": bool, "weeks": [W2, W1, W0]}`.
  - timeline·hadSecrets처럼 **매 호출 fresh 계산** (parent_reports 캐시에 저장 안 함 → 프롬프트 버전 갱신 불필요, LLM 0).
- 임계값은 상수로 분리: `PATTERN_MIN_RECENT = 3`, `PATTERN_MIN_DAYS = 4` (조정 가능하게 — 팀장 지시).

## 2. 표시 위치 — KiddyReportCard (키디의 한 주)

- 감정 타임라인/분포 **바로 아래**에 부드러운 배너 1장 (`patternSignal.active`일 때만).
- 톤: 위기 카드(💛 care_signal, 안전 탭)와 시각적으로 구분 — 경고색 금지, 리포트의 따뜻한 톤 유지 (예: 하늘색/보라 계열 소프트 카드).
- 읽음 처리·저장 없음 (리포트 열 때마다 현재 상태 반영 — 흐름이 좋아지면 자연히 사라짐).

## 3. 부모 문구 — ✅ 팀장 검수 확정 (verbatim, 임의 수정 금지)

> **본문 (원안 승인):**
> "최근 2주, {아이}의 마음에 흐린 날이 조금씩 늘고 있어요.
> 무슨 일인지 캐묻기보다, 함께하는 시간을 조금 늘려보시면 어떨까요?
> 키디도 매일의 안부에서 {아이}의 마음을 살피고 있을게요."
>
> **서브라인 (팀장 추가 — 작은 글씨, '진단 아님' 명시 필수 조항):**
> "이 신호는 진단이 아니에요. 그저 오늘, 조금 더 함께하자는 이야기예요."

- 서브라인은 본문 아래 작은 글씨(보조 톤 컬러)로 — 본문에 임상적 부정문을 넣으면 무거워지므로 분리 (팀장 지시).
- 금지: 진단 단어(우울·불안 장애 등), 미래 예측("괜찮아질 거예요"), 원인 추정("~때문일 수 있어요"), 1388 안내(그건 위기 카드 전용 — 층위 혼동 금지).
- {아이} 이름 조사 처리는 기존 josa 유틸 관례.

## 4. 검증

- **발화:** 해인 테스트 시드(주간 부정 0/2/4, 매일 체크인 — 삽입 완료) 리포트 → 배너+서브라인 표시 + weeks=[0,2,4]. (R 시드 "하늘"도 동일 패턴 — R에서 재검증)
- **미발화:** 평온 프로필 → 배너 없음 / 부정 0→1→2 → 없음 (W0<3) / **W1 체크인 3일 이하 → 없음 (데이터 하한)** — 하한 케이스는 단위 검산으로 확인해도 됨.
- **비저장:** parent_reports 행에 patternSignal 없음 (fresh 계산) / answers 미접근 (mood 컬럼만).
- **회귀:** 기존 리포트 필드 구조 diff 0 (additive만) / 캐시 히트·미스 동작 동일.
- **층위:** 위기(care_signal)와 동시 발생 시 둘 다 각자 자리에 — 서로 억제하지 않음.

## 5. 막히면 멈추고 보고

- 21일 조회가 기존 캐시 신선도 로직과 얽히면 → 임의 재설계 말고 보고.
- 임계값·문구 수정 필요 판단 → 제안으로 보고 (팀장 검수 대상).

## 6. 알려진 한계 — 고도화 노트 (v1 의도된 절제, 팀장 기록 지시)

- 이 규칙은 **'상승 추세'만** 잡는다 — `4→4→4`처럼 **지속적으로 높은 상태는 발화하지 않음**. v1에서는 의도된 절제(지속 상태는 감정 타임라인 자체에 보임). 고도화(9~11월)에서 **비율 기반 + 지속 고위험 규칙** 검토.
- 코드 주석에도 이 한계를 남길 것 (다음 작업자가 버그로 오인하지 않게).
