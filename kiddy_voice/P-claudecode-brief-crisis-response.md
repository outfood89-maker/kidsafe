# [Claude Code 작업 지시서] P — P0 2차: 위기 신호 대응 한 겹 (LLM 0, 사람이 검증한 고정 응답)

> **왜:** "아이가 키디에게 '맞았어', '죽고 싶어' 같은 말을 하면?" — 심사 Q&A 최다 예상 질문이자, 심사위원이 **직접 입력해볼 가능성이 높은** 시나리오(URL 체험 심사). 현재 답 없음(입력 필터 0, 시스템 프롬프트 한 겹뿐).
> **설계 원칙(팀장 확정):** 감지 시 LLM 건너뛰고 **사람이 검증한 고정 응답**만 / 부모에게 **부드러운 신호**(내용 미전달 — 존재만) / 전문 상담 안내 / **절대 금지: 진단 표현·자동 신고·위기 내용 저장**.
> **🚨 카피 게이트:** §3·§4의 문구는 **후보안** — 착수 전 팀장 검수 필수. 검수 결과로 교체 후 구현.
> **막히면 임의로 우회하지 말고 멈추고 보고.**

## 0. 확정 사실 (코드 검증됨) — 아이 자유 텍스트가 흐르는 경로 3개

| # | 경로 | 서버 도달? | 스크리닝 배치 |
|---|---|---|---|
| 1 | 챗봇 입력 → [chat.py `chat_with_kiddy`](../server/routers/chat.py#L99) | O (Claude 호출 전) | **서버** |
| 2 | 체크인 질문 답(그 외·STT) → [checkins.py `/react`·`/react/stream`](../server/routers/checkins.py#L374) | O (Claude 호출 전) | **서버** |
| 3 | **한 박자 더 후속답 STT 자유발화** → [DailyCheckin.jsx `selectFollowup`](../client/src/components/DailyCheckin.jsx#L402) | **X — 전부 로컬 템플릿, 서버 미경유** | **클라이언트** (서버만으론 구멍) |
- 슬픈·화난 기분(😢😡)의 받아주기도 로컬 고정 템플릿(Claude 안 씀) → 그 경로의 자유 텍스트도 경로 3과 같이 **클라 스크리닝** 필요.
- 기존 alerts 테이블([alerts.py](../server/routers/alerts.py))은 **영상 전용 스키마**(videoId·totalScore·thumbnail…) → 위기 신호에 재사용 부적합. **신호 전용 테이블 신설**(§4) — 텍스트 컬럼이 아예 없어 **구조적으로 내용 저장 불가**(발표 방어 포인트).

## 1. 위기 사전 (공유 상수 — 서버·클라 동일 내용 유지)

신규: `server/safety_lexicon.py` + `client/src/utils/safetyLexicon.js` (같은 내용, 주석으로 상호 참조 명시).

**2단계 구분 — 오탐(false positive)이 핵심 리스크:** 아이는 "게임에서 죽었어", "공룡이 죽었어"를 일상적으로 말한다.

```python
# HIGH — 감지 시: 고정 응답 + 부모 신호 생성. '나'를 주어로 한 명확한 위기 표현만.
HIGH_PATTERNS = [
    "죽고 싶", "죽을래", "죽어버리고 싶", "사라지고 싶", "없어지고 싶",
    "살기 싫", "태어나지 말", "자살",
    "맞았어", "나를 때", "나 때렸", "밟았어",  # 폭력 피해 (1인칭 방향)
]
# SOFT — 감지 시: 고정 응답만(알림 없음). 애매하거나 3인칭 가능성이 큰 표현.
SOFT_PATTERNS = ["무서워서 못 자", "아무도 날 안 좋아", "나만 미워", "혼자인 것 같"]
# EXCLUDE — HIGH 매치라도 이 문맥이면 통과 (게임·이야기 3인칭). 예: "게임에서 죽었", "공룡이 죽",
EXCLUDE_HINTS = ["게임", "공룡", "좀비", "몬스터", "캐릭터", "이야기에서", "동화에", "영상에서"]
```
- ⚠️ 목록은 **초기 최소셋** — 작업자가 임의로 늘리지 말 것(추가는 팀장 검수 대상). 매칭은 단순 `in` 부분일치 + EXCLUDE 우선.
- ⚠️ "죽었어"(과거·3인칭 흔함)는 HIGH에 **넣지 않는다** — "죽고 싶"(의지·1인칭)과 구분.

## 2. 감지 시 동작 (경로별)

**서버 (경로 1·2):** Claude 호출 **전에** 스크리닝 → HIGH/SOFT 매치 시 LLM 건너뛰고 고정 응답 반환(스트림 경로는 고정 텍스트를 스트림 형태로 흘려도 됨 — 프론트 변경 최소화). HIGH면 §4 신호 생성.
**클라 (경로 3 + 부정감정 자유텍스트):** 로컬 템플릿 대신 고정 응답 표시(+TTS). HIGH면 신호 생성 API 1회 호출(`POST /care-signals` — §4).
**공통:**
- 🚨 **위기 텍스트를 저장하지 않는다** — 체크인 answers에 위기 자유발화가 들어가는 경우, 해당 답은 `"[아이의 말은 키디가 들었어요]"` 같은 플레이스홀더로 치환하거나 비운다(내용 미저장 — O 브리프의 미저장 정책과 일관).
- 챗봇은 대화 저장 여부 확인 후 동일 원칙 적용(저장한다면 해당 메시지 마스킹).
- 감지 후에도 앱은 정상 흐름 유지(체크인은 계속 진행 가능, 챗봇은 다음 입력 가능) — 아이를 차단·격리하는 UX 금지.

## 3. 아이에게 주는 고정 응답 — 🚨 카피 후보 (팀장 검수 대기)

4~7세 언어 + 위로 원칙(미래 예측·약속 금지, 공감+곁에 있기, "어른에게 말해줘" 연결). **상담사 흉내·진단 금지.**

**후보 A — 자해·자살 신호 (HIGH):**
> "말해줘서 정말 고마워. 네 마음이 많이 아팠구나.
> 키디가 꼭 곁에 있을게.
> 그리고 이 마음은 엄마, 아빠나 믿을 수 있는 어른에게 꼭 말해줘. 키디가 부탁할게."

**후보 B — 폭력·아픔 신호 (HIGH):**
> "말해줘서 고마워. 많이 아프고 무서웠겠다.
> 이건 어른이 꼭 알아야 하는 일이야.
> 엄마, 아빠나 선생님한테 꼭 말해줘. 키디도 네 편이야."

**후보 C — SOFT (외로움·불안):**
> "그런 마음이 들 때가 있어. 말해줘서 고마워.
> 키디가 옆에 있을게. 오늘은 좋아하는 걸 하나만 같이 찾아보자."

## 4. 부모 신호 — 신설 테이블 + 리포트/대시보드 표시

**테이블 (SQL은 Freddie가 Supabase 대시보드 수동 실행 — 작업자는 SQL 준비):**
```sql
-- care_signals: 텍스트 컬럼이 없다 = 위기 '내용'은 구조적으로 저장 불가 (hadSecrets 철학의 연장)
create table care_signals (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null,
  profile_id uuid not null,
  level text not null default 'high',   -- 'high'만 우선 (soft는 신호 안 만듦)
  read boolean not null default false,
  created_at timestamptz not null default now()
);
```
- 서버: `POST /care-signals`(생성 — 경로 3용) + 경로 1·2는 서버 내부에서 직접 insert. `GET /care-signals?profileId=` (부모 조회). 같은 날 중복 신호는 1건으로(도배 방지).
- **부모 화면(ParentDashboard 알림 영역):** 🚨 카피 후보 (팀장 검수 대기):
> "오늘 아이의 말에서 어른의 관심이 필요한 신호가 있었어요.
> 무슨 일인지 캐묻기보다, 오늘은 아이 곁에서 가만히 들어봐 주세요.
> 마음이 쓰이시면 전문 상담과 이야기해보실 수 있어요 — 청소년·아동 상담전화 **1388** (24시간)."
- **금지:** 진단 단어(우울·자해위험 등), 위기 발화 내용/카테고리 세부 노출, "신고" 자동화. 신호는 **존재만** 전달 (비밀 원칙의 유일한 예외 지점이며, 예외조차 존재-신호 방식).
- 아이 화면에는 기관명·전화번호를 노출하지 않는다(아이에겐 "어른에게 말해줘"까지만) — 기관 안내는 부모 화면 전용.

## 5. 검증

**정탐:** 챗봇에 "죽고 싶어" 입력 → 고정 응답(후보 A) + LLM 미호출 + care_signal 1건 / 체크인 '그 외'에 "아빠가 때렸어" → 후보 B + 신호 / 후속답 STT로 위기 발화 → 클라 스크리닝 작동.
**오탐:** "게임에서 죽었어", "공룡이 죽는 영상 봤어" → **정상 반응**(위기 응답 아님) / "무서운 꿈 꿨어" → 정상 or SOFT.
**저장:** DB 어디에도 위기 발화 원문 없음 (daily_checkins·chat 저장물·care_signals 전부).
**신호:** 부모 대시보드에 신호 카드 표시, 같은 날 중복 1건, 읽음 처리.
**회귀:** 평범한 체크인·챗봇 대화는 기존과 완전 동일(스크리닝이 지연·오작동 유발 없음).
