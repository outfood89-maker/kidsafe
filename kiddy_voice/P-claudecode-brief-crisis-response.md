## 1. 위기 사전 (공유 상수 — 서버·클라 동일 내용 유지)

신규: `server/safety_lexicon.py` + `client/src/utils/safetyLexicon.js` (같은 내용, 주석으로 상호 참조 명시).

**2단계 구분 — 오탐(false positive)이 핵심 리스크:** 아이는 "게임에서 죽었어", "공룡이 죽었어"를 일상적으로 말한다. (✅ 2026-07-02 팀장 검수 반영: "맞았어"→"한테 맞았", "밟았어" 삭제, EXCLUDE 스코프 자해·죽음 계열 한정)

```python
# HIGH_SELF — 자해·죽음 계열. 감지 시: 고정 응답 A + 부모 신호. EXCLUDE 문맥이면 통과.
HIGH_SELF_PATTERNS = [
    "죽고 싶", "죽을래", "죽어버리고 싶", "사라지고 싶", "없어지고 싶",
    "살기 싫", "태어나지 말", "자살",
]
# HIGH_VIOLENCE — 폭력 피해 계열. 감지 시: 고정 응답 B + 부모 신호.
# ⚠️ EXCLUDE 적용 금지 — "게임하다가 아빠한테 맞았어"는 진짜 신호. (팀장 수정 2)
HIGH_VIOLENCE_PATTERNS = [
    "한테 맞았",   # 피해 방향 명확 ("퀴즈 맞았어"·"주사 맞았어" 오탐 방지 — 팀장 수정 1)
    "나를 때", "나 때렸", "때렸어", "때려서",
]
# SOFT — 감지 시: 고정 응답 C만(알림 없음). 애매하거나 3인칭 가능성이 큰 표현.
SOFT_PATTERNS = ["무서워서 못 자", "아무도 날 안 좋아", "나만 미워", "혼자인 것 같", "괴롭"]  # "괴롭"=따돌림 신호 (팀장 선택 권고 채택)
# EXCLUDE — ⚠️ HIGH_SELF에만 적용 (게임·이야기 3인칭 죽음 오탐 방지). HIGH_VIOLENCE·SOFT엔 미적용.
EXCLUDE_HINTS = ["게임", "공룡", "좀비", "몬스터", "캐릭터", "이야기에서", "동화에", "영상에서"]
```
- ⚠️ 목록은 **팀장 검수 확정셋** — 작업자가 임의로 늘리거나 빼지 말 것(변경은 팀장 검수 대상). 매칭은 단순 `in` 부분일치.
- ⚠️ 판정 순서: ① HIGH_VIOLENCE 매치 → 즉시 HIGH(폭력) — EXCLUDE 안 봄. ② HIGH_SELF 매치 → EXCLUDE 힌트 있으면 통과, 없으면 HIGH(자해). ③ SOFT 매치 → SOFT.
- ⚠️ "죽었어"(과거·3인칭 흔함)는 HIGH에 **넣지 않는다** — "죽고 싶"(의지·1인칭)과 구분. "맞았어" 단독도 **넣지 않는다**("퀴즈 맞았어!"·"주사 맞았어" — 우리 앱에 OX 퀴즈 있음).

## 2. 감지 시 동작 (경로별)

**서버 (경로 1·2):** Claude 호출 **전에** 스크리닝 → HIGH/SOFT 매치 시 LLM 건너뛰고 고정 응답 반환(스트림 경로는 고정 텍스트를 스트림 형태로 흘려도 됨 — 프론트 변경 최소화). HIGH면 §4 신호 생성.
**클라 (경로 3 + 부정감정 자유텍스트):** 로컬 템플릿 대신 고정 응답 표시(+TTS). HIGH면 신호 생성 API 1회 호출(`POST /care-signals` — §4).
**공통:**
- 🚨 **위기 텍스트를 저장하지 않는다** — 체크인 answers에 위기 자유발화가 들어가는 경우, **해당 답 항목을 answers에서 아예 제외하고 저장**한다(플레이스홀더 치환 금지 — O 브리프의 미저장 정책과 일관).
  - ⚠️ **플레이스홀더(예: "[아이의 말은 키디가 들었어요]") 사용 금지 이유 (팀장 보충, 2026-07-02):** 아이가 그 체크인을 공유(share=true)하면 플레이스홀더가 부모 리포트 하이라이트('나누고 싶어한 것')에 그대로 떠서 "뭔가 가려졌다"는 어색한 신호가 된다. **위기 마스킹된 답은 하이라이트 집계에 절대 도달하지 않아야** 하며, 부모에게 가는 위기 정보는 **care_signal 카드 하나로 단일화**한다.
- 챗봇은 대화 저장 여부 확인 후 동일 원칙 적용(저장한다면 해당 메시지 마스킹).
- 감지 후에도 앱은 정상 흐름 유지(체크인은 계속 진행 가능, 챗봇은 다음 입력 가능) — 아이를 차단·격리하는 UX 금지.

## 3. 아이에게 주는 고정 응답 — ✅ 팀장 검수 확정 (verbatim, 임의 수정 금지)

4~7세 언어 + 위로 원칙(미래 예측·약속 금지, 공감+곁에 있기, "어른에게 말해줘" 연결). **상담사 흉내·진단 금지.**

**응답 A — 자해·자살 신호 (HIGH_SELF) — 원안 승인:**
> "말해줘서 정말 고마워. 네 마음이 많이 아팠구나.
> 키디가 꼭 곁에 있을게.
> 그리고 이 마음은 엄마, 아빠나 믿을 수 있는 어른에게 꼭 말해줘. 키디가 부탁할게."

**응답 B — 폭력·아픔 신호 (HIGH_VIOLENCE) — 팀장 수정 반영 (가해자가 부모일 가능성 → "믿을 수 있는 어른"으로, 아동안전 표준 관행):**
> "말해줘서 고마워. 많이 아프고 무서웠겠다.
> 이건 어른이 꼭 알아야 하는 일이야.
> 선생님이나 믿을 수 있는 어른한테 꼭 말해줘. 키디도 네 편이야."

**응답 C — SOFT (외로움·불안) — 원안 승인:**
> "그런 마음이 들 때가 있어. 말해줘서 고마워.
> 키디가 옆에 있을게. 오늘은 좋아하는 걸 하나만 같이 찾아보자."

## 4. 부모 신호 — 신설 테이블 + 리포트/대시보드 표시

**테이블 (SQL은 Freddie가 Supabase 대시보드 수동 실행 — 작업자는 SQL 준비):**
```sql
-- care_signals: 텍스트 컬럼이 없다 = 위기 '내용'은 구조적으로 저장 불가 (hadSecrets 철학의 연장)
create table care_signals (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null,
  profile_id uuid not null,
  level text not null default 'high',   -- 'high'만 우선 (soft는 신호 안 만듦)
  read boolean not null default false,
  created_at timestamptz not null default now()
);
```
- 서버: `POST /care-signals`(생성 — 경로 3용) + 경로 1·2는 서버 내부에서 직접 insert. `GET /care-signals?profileId=` (부모 조회). 같은 날 중복 신호는 1건으로(도배 방지).
- **부모 화면(ParentDashboard 알림 영역):** ✅ 팀장 검수 확정 (원안 승인 + 셋째 줄 추가 — verbatim):
> "오늘 아이의 말에서 어른의 관심이 필요한 신호가 있었어요.
> 무슨 일인지 캐묻기보다, 오늘은 아이 곁에서 가만히 들어봐 주세요.
> 무슨 말이었는지는 키디도 전해드리지 않아요 — 아이가 직접 들려줄 수 있게요.
> 마음이 쓰이시면 전문 상담과 이야기해보실 수 있어요 — 청소년·아동 상담전화 **1388** (24시간)."
- **금지:** 진단 단어(우울·자해위험 등), 위기 발화 내용/카테고리 세부 노출, "신고" 자동화. 신호는 **존재만** 전달 (비밀 원칙의 유일한 예외 지점이며, 예외조차 존재-신호 방식).
- 아이 화면에는 기관명·전화번호를 노출하지 않는다(아이에겐 "어른에게 말해줘"까지만) — 기관 안내는 부모 화면 전용.

## 5. 검증

**정탐:** 챗봇에 "죽고 싶어" 입력 → 고정 응답 A + LLM 미호출 + care_signal 1건 / 체크인 '그 외'에 "아빠가 때렸어" → 응답 B + 신호 / **"게임하다가 형한테 맞았어" → HIGH 정탐**(폭력 계열엔 EXCLUDE 미적용 검증 — 팀장 지정 케이스) / 후속답 STT로 위기 발화 → 클라 스크리닝 작동.
**오탐:** "게임에서 죽었어", "공룡이 죽는 영상 봤어" → **정상 반응**(위기 응답 아님) / **"퀴즈 다 맞았어!" → 정상** / **"똥 밟았어" → 정상** (팀장 지정 케이스 2건) / "무서운 꿈 꿨어" → 정상 or SOFT.
**저장:** DB 어디에도 위기 발화 원문 없음 (daily_checkins·chat 저장물·care_signals 전부) / **위기 발화 포함 체크인을 share=true로 공유해도** 부모 리포트 하이라이트에 위기 답·플레이스홀더가 안 뜸(다른 정상 답만 표시) — 위기 정보는 care_signal 카드로만 (팀장 보충 케이스).
**신호:** 부모 대시보드에 신호 카드 표시, 같은 날 중복 1건, 읽음 처리.
**회귀:** 평범한 체크인·챗봇 대화는 기존과 완전 동일(스크리닝이 지연·오작동 유발 없음).
